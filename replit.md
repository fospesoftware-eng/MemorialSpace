# Overview

This project is a pnpm workspace monorepo using TypeScript, designed to be a full SaaS platform for cemetery management called MemorialSpace. It aims to provide five distinct user surfaces: a B2B marketing landing page, a B2C public site for grave search and obituaries, a customer dashboard, a B2B cemetery operator dashboard, and a SaaS super-admin console. The platform includes advanced features like an interactive map maker for cemetery operators and an AI-powered tool to convert physical cemetery maps into digital, clickable ones.

The business vision is to modernize cemetery management through a comprehensive digital solution, improving operational efficiency for cemeteries and enhancing accessibility for the public. The project's ambition is to become a leading SaaS provider in the cemetery management sector.

### Public Cemetery Map & QR Memorial Pages:
The public cemetery website (`/c/:slug`) ships with two visitor-discovery surfaces beyond the home/find-grave/marketplace pages:
- **Interactive map (`/c/:slug/map`)** — backed by `GET /api/c/:slug/map`, which returns every plot in the org plus a sanitized burial summary when occupied (name + year-only birth/death + photoUrl + memorialCode if a QR exists; same PII rules as find-grave — no full DOB/DOD, no notes). Frontend renders a section-grouped grid of color-coded plot cells (available/reserved/occupied/maintenance), with status-pill filters, section filters, and per-status counts. Clicking a plot opens a side panel showing burial summary, "View memorial page" CTA when a QR is set up, and an "Order memorial services for this plot" CTA that propagates a `for=NAME&plotRef=PLOT-NO` context to the marketplace. A `<ScanLine>` icon badges plots that have a memorial page set up.
- **QR memorial page (`/c/:slug/memorial/:code`)** — backed by `GET /api/c/:slug/memorial/:code`, which resolves the QR code scoped to the org's slug, joins to burial + optional `memorials` row, honors `memorials.isPublic=false` (404 publicly even when QR + burial exist), and updates `qr_codes.scanCount` + `memorials.viewCount` in the background (fire-and-forget so a stalled write never blocks the page for grieving visitors). Returns full DOB/DOD/burialDate/biography/photos array (parsed defensively from `memorials.photos` JSON-string and merged with `burials.photoUrl`) plus plot label/section/row. The page renders a hero portrait + lifespan + age, a thumbnail photo gallery (clickable to swap the hero photo), Born/Passed/Resting cards, the biography in `whitespace-pre-line`, and an "Honour their memory" CTA that links to the marketplace with the same order-context query string.
- **QR target rewiring** — `POST /api/qr-codes` now generates `${publicOrigin}/c/${orgSlug}/memorial/${code}` instead of the hardcoded `memorialspace.app/memorial/${code}`. `publicOrigin()` trusts `x-forwarded-proto` (defaulting to https), so QRs minted in dev encode the dev preview domain and QRs minted in prod encode the published domain — no env wiring needed. Falls back to a non-slugged URL if the org somehow has no slug.
- **Family-editable memorials with separated read/write credentials** — every QR carries TWO secrets: the 16-hex `code` (in the public URL, read-only) and a 6-digit numeric `editPin` stored in `qr_codes.edit_pin` (operator-visible only, never embedded in any URL or QR image). The public site exposes `PATCH /api/c/:slug/memorial/:code/edit` which validates the PIN from the request body or `x-edit-pin` header, returns 401 for wrong PIN and 409 for legacy QRs without a PIN yet, and atomically creates a stub memorial row + relinks the QR inside a transaction on first edit. The edit page (`/c/:slug/memorial/:code/edit`) gates the title/biography/photos form behind a PIN input field with friendly error copy ("That PIN doesn't match", "ask the cemetery to issue one"). Operator UI: the b2b QR Codes page (`/app/qr-codes`) renders each card with the PIN displayed inline (`KeyRound` icon + monospace tracking-widest), an **Issue missing PINs** button calling `POST /api/qr-codes/backfill-pins` (idempotent — only fills NULLs) for legacy rows, and the existing **Generate for all burials** button (`POST /api/qr-codes/bulk-generate`) which mints code+PIN+stub-memorial in a per-burial transaction so memorial/QR inserts can never split. The public memorial GET intentionally never returns the PIN (smoke-tested). This decoupling means a shared memorial URL can be forwarded freely to family without enabling vandalism — only the cemetery operator can authorise edits by handing out the PIN.
- **Per-memorial privacy modes** — each memorial row carries a `visibility` text column (`open` | `basic` | `private`, defaulting to `open`) that families select on the edit page and the public `GET /c/:slug/memorial/:code` enforces. The same 6-digit edit PIN doubles as the read PIN. **`open`** returns the full payload to anyone with the QR. **`basic`** always returns name + dates + plot, but strips `biography` to `null` and `photos` to `[]` until the PIN matches; the public memorial page renders a Lock card in place of "Their Story" with an unlock input that re-runs the query with the PIN. **`private`** returns a minimal stub (just `{visibility, locked: true, cemeteryName}`) with all other fields nulled until the PIN matches; the page renders a full-screen lock gate. PATCH accepts `visibility` via Zod enum and updates atomically. Security hardening (per architect review):
  - **Header-only PIN** — the unlock PIN is only accepted via the `x-edit-pin` request header, never query-string; this keeps the credential out of access logs, proxy logs, and observability traces (it doubles as the edit credential, so a leaked URL would also enable vandalism). The frontend `usePublicMemorial(slug, code, pin)` hook sends the PIN via fetch `headers` and react-query keys on the PIN string so locked/unlocked responses don't share a cache slot.
  - **Cache-poisoning prevention** — every memorial GET sets `Cache-Control: no-store, private` and `Vary: x-edit-pin` so no shared cache (browser, CDN, proxy) can serve an unlocked response to a later anonymous visitor.
  - **Rate-limit on wrong PIN reads** — wrong PIN reads hit the same `EDIT_PIN_MAX=5/EDIT_PIN_WINDOW_MS=15min` in-memory limiter as PATCH, keyed on `${ip}:${code}:read`. After 5 wrong attempts the GET returns 429 + `Retry-After`, so the 10^6 keyspace can't be brute-forced online. Correct PINs bypass the limiter and reset the bucket on success.
  - **Edit page form gate** — `memorial-edit.tsx` threads the typed PIN into the GET (so private memorials can hydrate with real content), but only renders the Privacy/Title/Bio/Photos/Save form once `memorial.locked === false`. While locked, only the PIN input is shown plus a "This memorial is set to Private/Basic — enter the PIN to load existing settings" notice, so an empty form can never overwrite real content.
  - **PATCH already 409s when `qr.editPin` is null**, so a memorial can't be set to `private`/`basic` without a working PIN to unlock it with.
  - Smoke-tested: open → full payload (cache headers correct), set basic → bio/photos stripped without PIN, with PIN → full, set private → fully nulled stub, query-string PIN ignored, 6 wrong header PINs → attempts 1-5 return 200 with locked stub then attempts 6+ return 429, correct PIN after rate-limit still unlocks (resets bucket), restore open → back to full.
- **Order context propagation** — when a visitor clicks "Order memorial services for this plot" (from the map) or "Browse memorial marketplace" (from a memorial page), the URL carries `?for=NAME&plotRef=PLOT-NO`. Marketplace pulls these off on first mount, persists them to `sessionStorage` under `cemetery-order-context:${slug}`, strips them from the URL via `history.replaceState` (so back/refresh doesn't repaint a stale banner), shows a dismissible "Ordering for a loved one" banner with a clear button, and the cart reads the same key on first mount to prefill `customerNotes` with `[For NAME — Plot PLOT-NO] ` (only when notes are empty, never overwriting visitor input). Operators see the context inline in the order's notes — no schema change required.
- **Navigation** — "Cemetery Map" sits between "Find a Grave" and "Marketplace" in the public nav (mobile and desktop). The home page gains a two-card "Visit the grounds / Memorial QR codes" section linking to the map and marketplace respectively, themed via the same `--site-*` CSS variables so all 3 themes apply automatically.

# System Architecture

The project is structured as a pnpm workspace monorepo, utilizing Node.js 24 and TypeScript 5.9.

## Core Technologies:
- **API Framework:** Express 5
- **Database:** PostgreSQL with Drizzle ORM
- **Validation:** Zod (`zod/v4`) and `drizzle-zod`
- **API Codegen:** Orval (from OpenAPI spec)
- **Build Tool:** esbuild (CJS bundle)

## UI/UX Decisions:
The platform features a premium dark green theme with a gold accent (`#d4a843`) reserved for the admin surface and marketing elements. Both light and dark color palettes are available, managed by a `ThemeProvider` that persists the user's choice to `localStorage`. Sidebar elements retain a dark theme regardless of the main content area's palette.

## Feature Specifications:

### Routing:
The application uses `wouter` for routing, with each of the five surfaces having its own base path (e.g., `/find/*`, `/account/*`). Cross-surface navigation is handled by prefixing paths with `~` (e.g., `~/find`), while internal navigation uses relative paths. Sign-in pages are themed and redirect to their respective dashboards via a full page reload.

### Cemetery Configuration:
Operators can manage Plot Types, Burial Spot Types, and a Map Background Library. These configurations are stored in `localStorage` using a `useStored<T>` hook, which ensures data persistence and cross-component/tab synchronization. The `setStored` function handles persistence failures gracefully.

### Plot Map Maker:
A full-screen editor (`/app/map-maker`) allows cemetery operators to create and manage digital maps.
- **Tools:** Select, Plot (rectangular plots with customizable types), Circle (drag-from-center radial plots for ponds/gardens/circular features), Path (flexible polyline for paths/roads/bridges), and Spot (pins with lucide icons for individual burial spots, including dossier panels for details and image uploads). Each spot supports up to `MAX_HEADSTONE_IMAGES` (8) headstone photographs (front, back, inscription close-up, surrounding area, etc.) — the file input accepts `multiple`, downscales each pick to 400px webp/jpeg in parallel via `Promise.allSettled` (partial-failure tolerant), and merges via a functional `setDoc` so concurrent batches can't clobber each other. The legacy single-string `headstoneImage` field is migrated into the first slot of `headstoneImages` by `migrateDoc()`. The SpotEditor renders a 3-column thumbnail gallery with per-image remove buttons and an "x / 8" counter.
- **Path tool (flexible roads / pathways):** A click-to-place polyline drawer for any feature that doesn't fit a rectangle — winding paths, roads, bridges, etc. Each click appends a vertex (capped at `MAX_PATH_VERTICES = 200`), the cursor draws a dashed "rubber band" preview from the last vertex, and **double-click**, **Enter**, or **Esc** finishes (Esc to cancel; tool switches mid-draft commit if usable, otherwise discard). **Backspace** during drawing pops the last vertex. Hotkey **P**. The plot is stored as `pathPoints: [x,y][]` plus `pathWidth: number` (default `DEFAULT_PATH_WIDTH = 14`); the bounding box (x/y/w/h) is auto-derived with half-stroke padding so move/select hit-testing still works. Render is a stroked open `<polyline>` with rounded caps/joins, an invisible `pathWidth + 8`px hit-strip on top so narrow roads remain easy to click, and a dashed selection halo behind the body. The `PlotEditor` swaps W/H/X/Y for a single "Path width" range slider (2–60 px) that recomputes the bbox on change. The corner resize handle is hidden for path plots (path scaling happens via the slider). `migrateDoc()` coerces/clamps the new fields and drops the path entirely if fewer than 2 valid vertices remain.
- **Layout:** The editor renders outside the main B2B layout to utilize the entire viewport, with collapsible side panels and a top toolbar for navigation, zoom, and display options.
- **Backgrounds:** Supports image uploads (downscaled to 1600px webp), drag-and-drop, and selection from a recent library. Uploads are added to the background library automatically.
- **Interactivity:** Includes 2D/3D view toggle, zoom functionality, "Fit to screen" option, save/load multiple maps to `localStorage`, JSON export, and keyboard shortcuts (V/R/C/P/S/H for Select/Plot/Circle/Path/Spot/Pan tools, Delete/Backspace, Esc, F for fullscreen). The **Pan (Hand) tool** scrolls the canvas viewport on drag — useful for navigating dense imported maps without accidentally selecting plots. **Plot labels** are off by default (toggle with the toolbar "Labels" button); hovering any plot momentarily reveals its label with a white halo for legibility, and the selected plot's label is always shown.
- **Circle tool (radial plots):** Drag from the press point (treated as the center) outward; the cursor distance defines the radius. Stored as `circle: {cx,cy,r}` plus a square bounding box (`x = cx-r, y = cy-r, w = h = 2r`) so move/select hit-testing works unchanged. Hotkey **C**. Shares the same draft ref as the rect tool (only one in-progress shape at a time) — the SVG draft preview branches on `draftRect.circle`. Render priority is `path > circle > polygon > rect` for both the visible shape and 3D shadow. The corner resize handle is hidden for circles; the `PlotEditor` swaps W/H/X/Y for a Radius slider (4–2000 px, matching `migrateDoc` clamp) plus Center X/Y inputs that recompute the bbox on change. `migrateDoc()` validates and clamps the field (drops the circle if `r < 1` or `r > 2000`).
- **Polygon tool (irregular filled shapes):** Click-to-place vertices forming a **closed** filled shape — the perfect tool for non-rectangular cemetery sections, plazas, and irregularly shaped land features. Each click appends a vertex (capped at the same `MAX_PATH_VERTICES = 200` as the path tool); a translucent fill preview updates live as the cursor moves, with two rubber-band edges (last-vertex → cursor and cursor → first-vertex) so the user always sees the shape that will be committed. **Double-click**, **Enter**, or switching tools commits; **Esc** cancels; **Backspace** pops the last vertex. Requires ≥ 3 vertices to commit (a triangle is the smallest valid polygon). Hotkey **G** (for "polyGon" — P was already taken by Path). Stored as the existing `points: [x,y][]` field (the same one AI-imported polygon plots use), so render, hit-test, move, and migration code paths are reused unchanged. Bbox is the tight min/max of the vertices (no stroke padding — the fill stays inside the geometry). Shares the `draftPath` infrastructure with the Path tool via a `closed: boolean` discriminator on the draft state — both are click-based, both use the dblclick/Enter/Esc/Backspace gestures, and both auto-commit-or-discard on tool change.
- **Per-type default shape:** Every `PlotType` carries an optional `defaultShape: "rect" | "circle" | "polygon" | "path"` (defaults to `"rect"` when missing). The Cemetery Setup table exposes a **Shape** column (Select with icon + hint) per row; the Map Maker palette button reads the active type's `defaultShape` and auto-switches the canvas tool via a `shapeToTool()` helper when clicked. Default seeds: PATH/BR=path, WTR/GDN=circle, all other defaults=rect. The user can still pick any tool manually from the toolbar.
- **Plot Type Palette:** The default palette covers cemetery sections (RC, CON, FC, MU) plus infrastructure features (PATH road, BLD building, COL columbarium, MAU mausoleum, WTR lake/water, BR bridge, GDN garden, PRK parking, GT gate). Each type has a distinct fill/stroke colour and a sensible `defaultShape`. New defaults ship via a per-version `PLOT_TYPE_MIGRATIONS` map keyed by `PLOT_TYPES_SEED_VERSION` so existing users get only the *newly added* ids merged into their stored list (deletions of older defaults are preserved, customisations are never overwritten).
- **Implementation:** Uses `svg.getScreenCTM().inverse()` for accurate pointer math, a `useRef` for drag state management, and `lucide` icons embedded as SVG children. It includes a `migrateDoc()` function for backward compatibility with legacy map data and supports polygon outlines for plots.

### Cemetery Type & Columbarium Module:
Each organization is tagged with a `cemeteryType` chosen at signup — one of `church`, `private`, `pet`, `municipality`, or `columbarium` — stored on the `organizations` table along with a `featuresColumbarium` opt-in flag (always true when type is `columbarium`; opt-in for `church`). The signup dialog (`/app/organizations`) presents the five types as a Select with descriptive icons, exposes the columbarium checkbox conditionally for church organizations, and renders both as badges on each org card.

The standalone Columbarium module (`/app/columbarium`) is a list-and-detail experience for cinerary niche walls:
- **Schema:** `columbaria` (id, orgId FK cascade, name, description, rows, cols) + `niches` (id, columbariumId FK cascade, row, col, occupantName, dob, dod, inscription, photoUrl, status [available/reserved/occupied], notes) with `unique(columbariumId,row,col)`.
- **API:** REST under `/api/columbaria` — full CRUD on walls plus a position-based niche upsert endpoint `PUT /api/columbaria/:id/niches/:row/:col` (and matching `DELETE`) so the UI can write a niche by its grid coordinates without tracking server ids.
- **List view:** Grid of wall cards with capacity totals; "New wall" dialog with rows/cols inputs (creates a fully empty grid).
- **Detail view:** Live-updating stats (total/occupied/reserved/available), a 2D SVG-style niche grid with status colours (slate=available, amber=reserved, green=occupied) and initial portraits or uploaded photos at the top of each tile, name and lifespan at the bottom. Clicking a niche opens a Sheet for editing (occupant name, DOB, DOD, inscription, status, notes, photo upload). Photo uploads are downscaled client-side to 600px JPEG via `createImageBitmap` and stored as data URLs.
- **Creative 3D view:** Toggle button switches the wall into perspective mode using CSS `perspective: 1500px` on a parent container plus `rotateX()` and `scale()` on the grid, with a tilt slider (20–75°, default 45°). Reuses the same shadow-tile technique as the map maker's 3D mode, giving operators an immediate "wall standing in front of you" preview. A `ResizeObserver` auto-fits the wall to the available container width.

### Mausoleum Module:
The standalone Mausoleum module (`/app/mausoleum`) mirrors the Columbarium pattern but for above-ground casket structures. Sidebar entry sits beneath Columbarium under Cemetery Operations, using the `Building2` icon.
- **Schema:** `mausoleums` (id, orgId FK cascade, name, description, location, rows, cols) + `mausoleum_crypts` (id, mausoleumId FK cascade, row, col, `cryptNumber`, `cryptType` [single/companion/family], status [available/reserved/occupied], primary occupant fields, **second occupant fields** [`secondOccupantName`/`secondDob`/`secondDod`] for companion + family crypts, `inscription`, `photoUrl`, **owner record** [`ownerName`/`ownerContact`] for right-of-entombment tracking, `notes`) with `unique(mausoleumId,row,col)`.
- **API:** REST under `/api/mausoleums` — full CRUD on buildings plus position-based crypt upsert/delete `PUT|DELETE /api/mausoleums/:id/crypts/:row/:col`. **Every by-id endpoint is org-scoped** the same way the Accounting module is: a required `?orgId=` query param (400 if missing) plus `WHERE id=? AND organizationId=?` in every read/write/delete so cross-tenant access returns 404 (mausoleum routes) or no-op 204 with no actual delete (crypt delete uses `EXISTS` against the parent's org). Crypt delete intentionally returns 204 even on no-match so existence cannot be probed across tenants. **Concurrency-safe writes:** the crypt upsert and the mausoleum resize both run inside a single `db.transaction` with `SELECT ... FOR UPDATE` on the parent row before bounds/orphan checks and the write — closes the TOCTOU window where a concurrent resize could let an out-of-bounds crypt slip in or a concurrent upsert could land after orphan-check. Atomic upsert via `ON CONFLICT DO UPDATE` on the unique index, with `xmax = 0` on the returning clause to distinguish 201 (created) from 200 (updated) without a second round-trip. Resize without `force=true` refuses if any out-of-bounds crypts are non-empty (409 with `occupiedCount` + hint); empty cells are pruned either way.
- **List view:** Card grid of mausoleums with capacity totals + location chip and description; "New mausoleum" dialog with name/location/description/rows/cols inputs (rows 1–20, cols 1–40, total = rows × cols).
- **Detail view:** Live stats (total/occupied/reserved/available), 2D SVG marble wall with tier labels (T1 = bottom — operators read the wall the way they walk it) and position numbers across the top, status colours (slate=available, amber=reserved, green=occupied), companion/family crypts tagged with a cyan top stripe and a divider showing both occupants on a single tile (initial-letter portraits when no photo). Crypt tiles default to a `T{tier}-P{pos}` address label when no human-given crypt number is set.
- **Crypt editor sheet:** Opens on tile click. Per-crypt fields: crypt number, type (single/companion/family), status, portrait (downscaled client-side to 600px JPEG via `createImageBitmap`), primary occupant (name/DOB/DOD), conditional second-occupant block (only shown for companion/family — uses an `animate-in slide-in` reveal), inscription, owner record block (name + contact for right-of-entombment), and internal notes. The form `useEffect`-resets keyed on `existing` + position so opening a different crypt always shows fresh data.
- **Interior 3D view:** Distinct from the columbarium's top-down 3D — uses `rotateX(-tilt)` on the wall to give a "standing in the chapel looking up" perspective, with a darker floor band rendered into the SVG that reads as the floor when tilted. Tilt slider (5–60°, default 35°). Otherwise reuses the columbarium's shadow-tile depth technique and `ResizeObserver` auto-fit.

### Cemetery Map (read-only view):
The B2B `/map` page renders a grid of plots from `GET /api/plots/map/:orgId` and opens a Sheet on click. Beyond the plot facts (status badge, type, owner, price, notes), the Sheet **lazy-loads burial records** for the clicked plot via `useListBurials({ organizationId, plotId })` (gated on `enabled: selectedPlotId != null`, with an explicit `queryKey` from `getListBurialsQueryKey` to satisfy the generated UseQueryOptions type). Each burial card shows the deceased's photo (or an `ImageOff` placeholder), full name, DOB–DOD with computed age, religion badge, interment date, and notes — plus the plot/record IDs for traceability. Empty plots render a dashed "No burial records linked" state, loading shows two skeleton cards.

### Accounting Module:
A complete invoicing and accounts-receivable module under `/app/accounting/*`, scoped per organization, contract-first via OpenAPI + Orval codegen.
- **Schema:** Five new Drizzle tables in `lib/db/src/schema/`:
  - `customers` (org-scoped billing contacts: name, email, phone, address, notes)
  - `taxRates` (org-scoped, `ratePercent: real`, `isDefault`, `isArchived` for soft-delete)
  - `invoices` (`status: draft|issued|partially_paid|paid|voided`, `invoiceNumber: INV-YYYY-NNNN` allocated on issue, `subtotal/taxTotal/total/amountPaid: real`, `issueDate/dueDate/paidDate`, `notes`)
  - `invoiceItems` (FK invoice cascade, `description`, `quantity`, `unitPrice`, optional `taxRateId` snapshotting `taxRatePercent` so historical lines stay correct if rates change later)
  - `payments` (FK invoice cascade, `amount`, `paymentDate`, `method: cash|check|card|bank_transfer|other`, `reference`, `notes`)
- **Money:** all amounts stored as `real` USD; quantities as `integer`. Server-authoritative totals — clients never trust their own math.
- **API:** REST under `/api/{customers,tax-rates,invoices,payments,accounting}`. **Every endpoint is org-scoped**: list/by-id GETs and DELETEs require `?organizationId=` (returns 400 if missing), and every row lookup AND-filters on `organizationId` so a row in another org returns 404 rather than leaking. Create/update bodies require `organizationId` in the payload.
  - Customers/tax-rates: list (with `?includeArchived` for tax rates), create, get, update, delete (tax-rate delete = archive). Customer delete cascades to that customer's invoices via `onDelete: 'cascade'` on `invoices.customer_id`.
  - Invoices: list (with `?status=` filter), get-with-items+payments, create-draft, update-draft (only when `status=draft`), `POST /:id/issue` (allocates next `INV-YYYY-NNNN` and stamps `issueDate`), `POST /:id/void` (blocked on `draft`, `paid`, and already-`voided`). Cross-org tax-rate references are rejected with 400 (`TaxRateValidationError`) on both create and update by validating every referenced `taxRateId` belongs to the invoice's org via `inArray`.
  - Payments: list `?invoiceId=`, create, delete. Each create/delete recomputes the parent invoice's `amountPaid` and transitions the status (`issued ↔ partially_paid ↔ paid`) and `paidDate` automatically.
  - Accounting: `GET /accounting/summary` returns `{totalOutstanding, totalOverdue, paidThisMonth, invoicedThisMonth, draft/issued/paid/overdueCount, aging: [Current/1-30/31-60/61-90/90+]}` computed from open invoices vs `dueDate`.
- **Server helpers:** `artifacts/api-server/src/lib/accounting.ts` centralizes `computeInvoiceTotals()` (sums lines + per-line tax to 2 decimals; throws `TaxRateValidationError` on cross-org tax rates), `generateInvoiceNumber()` (MAX-suffix scan of current-year `INV-YYYY-%`, padded 4-digit), `recomputeInvoicePayments()` (sum payments → derive status + paidDate), and `loadInvoiceWithItems()` (one query for nested response). All routes use `BodyType<>` from Orval-generated `@workspace/api-zod` schemas for input validation.
- **Concurrency model for invoice issuing:** `POST /invoices/:id/issue` runs in a `db.transaction` and uses a status-guarded `UPDATE ... WHERE id=? AND organization_id=? AND status='draft' RETURNING *`. If the UPDATE matches 0 rows, a concurrent caller already issued the same invoice → return **409**. A unique index `invoices_org_invoice_number_unique` on `(organization_id, invoice_number)` plus a retry-on-23505 loop (max 8 attempts, recomputes the next number each time) handles the rarer race of two different drafts in the same org+year computing the same MAX+1.
- **Frontend pages** (`artifacts/memorial-space/src/pages/b2b/accounting/`):
  - `overview.tsx` — KPI cards (outstanding/overdue/paid this month/invoiced this month), AR aging buckets, recent-invoices table.
  - `customers.tsx` — searchable table + Dialog form for create/edit; AlertDialog confirm on delete.
  - `tax-rates.tsx` — table with active/archived states, Dialog form, default-rate switch; archive instead of delete.
  - `invoices-list.tsx` — table with status filter and search.
  - `invoice-edit.tsx` — full-page editor with line-item table (description/qty/unit price/tax rate select), live preview totals (server recomputes on save). Reused for new draft and edit-draft (gated by `?status=draft`).
  - `invoice-detail.tsx` — printable invoice body (CSS `print:` utilities hide chrome) with status badge, customer block, item table, totals, balance due. Action bar: Print, Edit/Delete (drafts), Issue (drafts), Record Payment (issued/partially_paid), Void (issued+). Inline payment history with delete.
- **Shared utilities:** `accounting/_shared.ts` exports `ORG_ID`, `formatMoney()`, `invoiceStatusBadgeClass()`, `invoiceStatusLabel()`.
- **Sidebar:** New "Accounting" group in `b2b-layout.tsx` with Overview, Invoices, Customers, Tax Rates entries (icons: BarChart3, Receipt, UserSquare2, Percent).
- **Smoke-tested end-to-end:** customer + tax-rate creation → 2-line invoice (sub $4,300, tax $58, total $4,358) → issue (`INV-2026-0001`) → partial payment $2,000 (status `partially_paid`) → final payment $2,358 (status `paid`, `paidDate` stamped) → AR summary reflects $4,358 paid this month.

### Cemetery Website Builder:
A full public-facing website builder so each cemetery operator can publish a branded site at `/c/:slug` with grave search, marketplace (services + products), and order intake — all org-scoped and tenant-isolated.
- **Schema** (`lib/db/src/schema/cemeterySites.ts`): four new tables, all with `organizationId` FK + `unique(organizationId)` or `unique(organizationId, …)` constraints:
  - `cemetery_sites` (one row per org: `theme` [classic-marble | modern-minimal | heritage-garden], `siteTitle`, `tagline`, `heroHeadline/Subheadline/ImageUrl`, `aboutText`, `contactPhone/Email/Address`, `openingHours`, `primaryColorOverride`, `isPublished` flag).
  - `cemetery_categories` (per-org product categories: `name`, `slug`, `displayOrder`, `iconName`).
  - `cemetery_products` (per-org marketplace items: `name`, `slug`, `description`, `priceCents`, `currency`, `categoryId` FK, `productType` [product | service], `photos: jsonb` [array of URLs], `isFeatured`, `isPublished`, `stripeEnabled` [v1 placeholder — UI shows "Coming soon"]).

### Team & Access Module:
A dedicated team-management surface at `/app/team` (members) and `/app/team/roles` (permission matrix), replacing the old single-page `/app/users` under Settings. Lives in its own sidebar group above Settings.
- **Schema** (`lib/db/src/schema/users.ts`): `users` table now carries a 5-value role enum (`owner | admin | manager | staff | viewer`), a 3-value status enum (`active | invited | suspended`), plus `jobTitle`, `phone`, `lastActiveAt`, `invitedAt`. `organizationId` is `NOT NULL` with `onDelete: cascade`, and a `unique(organizationId, email)` index enforces email uniqueness *per org* (so two cemeteries can both have an `info@` user).
- **API** (`artifacts/api-server/src/routes/users.ts`): every endpoint is org-scoped (`?organizationId=` required, AND-filtered on every read/write/delete so cross-tenant requests return 404). `POST /users` rejects duplicate emails per org both at the app layer and via the DB unique index. `POST /users/:id/resend-invite` bumps `invitedAt` and resets status to `invited`. `GET /team-summary?organizationId=N` returns `{total, byRole, byStatus}` for the dashboard cards.
- **Last-owner protection** (concurrency-safe): `PUT /users/:id` (role demotion or owner suspension) and `DELETE /users/:id` for an owner both run inside a `db.transaction` with `SELECT … FOR UPDATE` on the target row. Inside the lock, the owner-count is recomputed and a custom `LastOwnerError` is thrown → translated to a 409 `Cannot demote/suspend/remove the last owner — promote another member to owner first`. Closes the TOCTOU window where two concurrent owner-demotions could both pass the count check and leave the org with zero owners.

Invite + edit Dialogs share a Zod-style local validator and surface server errors via toast.
- **Frontend roles page** (`pages/team/roles.tsx`): five role overview cards with icons (Crown / ShieldCheck / Shield / Briefcase / Eye) showing capability counts and descriptions, plus a full permission matrix grouped into six capability families (cemetery operations, memorial services, public site & marketplace, accounting, team & access, account & billing). ### AI Map Maker:
A sub-feature within the Map Maker that converts raster or vector cemetery maps into digital, clickable maps.
- **Pipeline:**
    1.  **Upload:** Accepts image (`JPG/PNG/WebP`) and PDF files, downscaling to 1600px webp. Multi-page PDFs support page selection.
    2.  **Analyze:** POSTs image data and plot/spot types to `/api/ai/detect-map`.
    3.  **Review & Open:** Displays detected polygons and cells for review.
    4.  **Hand-off:** Saves the generated `MapDoc` to `localStorage` and redirects to the Map Maker.
- **Detection Modes:**
    -   **Grid Plan:** Triggered by vector PDFs with extracted text (`pdfText`). The client extracts text positions via `pdfjs.getTextContent()` and renders the page at 2400px so the small plot rectangles stay separable. The server (`lib/gridDetect.ts`) thresholds dark pixels, runs 4-conn BFS on the light interior regions to find candidate plot rectangles, drops the largest edge-touching component as the page background (other edge-touching plots are kept), filters by size/area/fill/aspect, applies an **outer-boxes-only filter** (drops any plot whose bbox is contained inside a strictly larger plot's bbox, with a small tolerance — keeps section containers and discards the small grave cells inside them), and matches each remaining plot to PDF text words by spatial containment in a 32×32 bucket grid (multi-grave sections aggregate their inner graves' names into the section label). Pixel-perfect outlines, no AI vision call. **Fallback**: if grid finds <10 plots (`MIN_GRID_PLOTS_TO_TRUST`) the route falls through to the colour-section pipeline rather than returning empty, so colour-section PDFs that happen to contain some text still work.
    -   **Color Section:** Used for raster images without `pdfText`. Employs a two-stage process:
        -   **Stage 1 (Computer Vision):** Decodes, downscales, and extracts color palettes. It performs per-color section detection (using tolerant masks, dilation, BFS, boundary tracing, and Douglas-Peucker simplification) and grave-cell detection within sections.
        -   **Stage 2 (Claude Classification):** Utilizes Anthropic's Claude model (`claude-sonnet-4-6`) to classify the detected color sections based on visual cues and operator-defined plot types.
- **Constraints:** Optimized for flat-color cemetery plans. Rate-limited to 12 detections per 5 minutes per IP, with a body limit of 12 MB for uploads.

## Authentication & Security

End-to-end real authentication shipped May 2026, replacing the earlier "open API + mock sign-in" demo posture.

- **Backend** (`artifacts/api-server/src/lib/auth.ts`, `routes/auth.ts`):
  - `POST /api/auth/login` accepts `{ email, password, kind }` where `kind ∈ {"cemetery","family","admin"}`. Cemetery/family sign-in checks `usersTable.passwordHash`; admin sign-in checks `platformAdminsTable.passwordHash`. Both use bcrypt cost-12 with constant-time compare.
  - `POST /api/auth/logout` destroys the session and clears the `ms.sid` cookie.
  - `GET /api/auth/me` returns the current `SessionUser` or 401.
  - Sessions are cookie-backed (`httpOnly`, `sameSite=lax`, `secure` in production, 14-day rolling) and stored in Postgres via `connect-pg-simple` against the Drizzle-managed `session` table. `SESSION_SECRET` is required at startup.
  - Middleware: `requireAuth` (any session), `requirePlatformAdmin` (admin tier), `requireOrgUser` (cemetery tier), `enforceOrgScope` (forces `organizationId` on query/body for cemetery users so a forged `?organizationId=X` cannot cross tenants).
  - Router-level guards in `routes/index.ts`: only `health`, `auth`, `publicApi` (`/public/*`), and `cemeterySites` (which serves the public `/c/:slug/*` cemetery websites) are unauthenticated. All B2B routers (`organizations`, `users`, `plots`, `burials`, `bookings`, `memorials`, `workOrders`, `qrCodes`, `obituaries`, `marketplace`, `dashboard`, `columbaria`, `mausoleums`, `customers`, `taxRates`, `invoices`, `payments`, `accounting`) are mounted behind `requireOrgUser`. The AI map detection endpoint is mounted behind `requireAuth`. Admin routes use their own `requirePlatformAdmin`.
  - **Tier verification**: `kind` from the client login payload is treated as a hint only. The server derives the actual tier from `usersTable.role` (`owner|admin|manager|staff` → cemetery, `viewer` → family) and 403s if the requested tier doesn't match the role, so a B2C account cannot escalate by hitting the cemetery sign-in route.
  - **Session fixation defense**: `req.session.regenerate()` is awaited before attaching the user on every successful login (admin, cemetery, family).
  - Rate limit: `express-rate-limit` 20 req / 15 min on `/api/auth/login`, with `skipSuccessfulRequests` so honest users aren't punished.
  - `helmet` for default security headers; global error handler returns `{error}` JSON without stack traces.
- **Known gaps (not yet implemented)**: per-account login lockout/backoff (only IP-level limiter today); CSRF tokens for cookie-authenticated writes (relying on `SameSite=Lax` + same-origin proxy for now); password complexity policy and password-reset flow; per-tenant scoping on individual `:id` lookups in some routers (currently relies on `enforceOrgScope` rewriting `organizationId` query/body and the route handlers using it — direct `WHERE id = ? AND organizationId = ?` should be added defensively to every by-id read/update/delete in a follow-up pass).
- **Frontend** (`artifacts/memorial-space/src/lib/auth.tsx`, `pages/auth/sign-in-form.tsx`, `App.tsx`):
  - `AuthProvider` + `useAuth()` + `RequireAuth` component. All `fetch` calls in admin & app dashboards include `credentials: "include"`.
  - All three sign-in surfaces (`/sign-in/cemetery`, `/sign-in/family`, `/admin/sign-in`) submit to the real backend with the appropriate `kind` and route the user to the matching dashboard on success.
  - `/app/*`, `/admin/*`, and `/account/*` route trees are wrapped in `RequireAuth`; on 401/403 the admin api client redirects to the admin sign-in.
- **Seed** (`pnpm --filter @workspace/scripts run seed-auth`): idempotent. Hashes any existing demo users that lack a `passwordHash`, ensures one platform admin (`admin@memorialspace.com / SuperAdmin2026!`), one cemetery operator (`ops@riversidememorial.com / Cemetery2026!`), and one family demo user (`sarah.chen@email.com / Demo2026!`).
- **Smoke test**: noauth admin metrics → 403, valid login → 200 + cookie, authed admin metrics → 200, `/me` returns the right tier, bad creds → 401.
